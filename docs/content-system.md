# Content System
## rebuilttheworld.org
### Version: v0.1

---

## Purpose

This document defines the initial content system of `rebuilttheworld.org`.

Its purpose is to make content creation more structured, consistent, and durable across the site.

It clarifies:

- what kinds of content exist in the system
- how they differ from each other
- how they should be written
- how they should be organized
- how content should remain readable over time

This is a foundational content design document, not a final publishing manual.

---

## Core Principle

Content on `rebuilttheworld.org` should not be treated as isolated pages.

It should function as part of a broader system of:

- orientation
- meaning
- structure
- preservation
- continuity
- long-term readability

The content system must support both present use and future understanding.

---

## Content Model Overview

The site should organize content into clearly defined layers.

### Layer 1 — Orientation Content
Short, stable, public-facing content that helps people understand where they are.

Examples:
- homepage intro
- section intros
- one-line definitions
- short positioning statements
- entry guidance

### Layer 2 — Interpretive Content
Content that explains ideas, systems, relationships, and meaning.

Examples:
- About page
- Manifesto
- System page
- LAB overview
- project descriptions

### Layer 3 — Living Content
Content that reflects movement, experimentation, and active thought.

Examples:
- Signals
- short notes
- research fragments
- process updates
- public traces of development

### Layer 4 — Preservation Content
Content meant to be durable, referenceable, and archivable.

Examples:
- field notes
- releases
- documentation records
- archive entries
- versioned materials
- structured public documents

---

## Content Goals

All site content should aim to be:

- clear
- structured
- meaningful
- navigable
- durable
- reference-aware
- readable by future audiences

Content should not exist only to fill space.

Every visible page or section should have a definable role in the system.

## Primary Content Types

The following content types should form the core system.

### Role

### Examples
- what RTW is

- stable wording
- low volatility
- high clarity

- About page definitions
- section intros
- module intros

- project summaries

## Content Depth Levels

The system should distinguish content by depth.

### Level A — Surface
Fast orientation, minimal friction.

Examples:
- hero copy
- one-line descriptions
- section labels
- short intros

### Level B — Mid-depth
Interpretive and explanatory.

Examples:
- About
- System
- project overviews
- manifesto sections

### Level C — Deep
Long-form, layered, or reference-heavy.

Examples:
- archive documents
- field notes
- public methods
- releases
- structured essays

A page should know which depth level it belongs to.

Not every page should try to do all three.

---

## Writing Principles

All content should follow these principles:

### 1. Clarity before performance
The text should communicate before it tries to impress.

### 2. Meaning before ornament
Language should carry content, not decoration.

### 3. Structure before volume
It is better to write less with clear structure than more without shape.

### 4. Durability before trend
The writing should remain understandable later.

### 5. Signal before noise
Only include text that helps orientation, meaning, or preservation.

### 6. Precision without sterility
The language can be exact while still feeling human.

---

## Tone and Style Requirements

Content across the site should feel:

- clear
- calm
- deliberate
- intelligent
- structured
- readable
- durable

Content should avoid:

- startup clichés
- hype language
- vague visionary filler
- inflated claims
- excessive abstraction
- unexplained jargon
- decorative complexity

The writing must remain aligned with `docs/brand-core.md`.

## Recommended Page Template

Important pages should follow a shared structure.

### Standard Page Pattern
1. Title  
2. One-line definition  
3. Purpose  
4. Context  
5. Main content  
6. Related modules  
7. Version  
8. Archive references  

This pattern should apply to major public-facing pages wherever appropriate.

---

## Recommended Project Entry Template

For project/module pages, use this pattern:

1. Project title  
2. One-line project definition  
3. Role in RTW  
4. Why it exists  
5. Current scope  
6. Related modules  
7. Status  
8. Version / update note  

This helps connected modules remain understandable in a shared system.

---

## Recommended Signal Template

Signals should remain lightweight but consistent.

### Suggested format
- Title
- Date
- Type
- Short content body
- Related area
- Optional next direction

### Example types
- note
- update
- fragment
- breadcrumb
- release
- observation

Signals should not become chaotic micro-posts.

Even lightweight content should stay structurally readable.

---

## Recommended Archive Template

Archive-oriented items should include stronger metadata.

### Suggested format
1. Title  
2. Date  
3. Version  
4. Type  
5. Summary  
6. Main content or linked file  
7. Related documents  
8. Preservation notes / references  

This supports long-term navigation and reuse.

---

## Metadata Requirements

Where relevant, content should include metadata such as:

- title
- date
- version
- type
- status
- related module
- related documents
- update note

Not all page types need all metadata, but archive-oriented content should include more of it.

---

## Content Status Logic

Content should eventually distinguish between types of maturity.

### Suggested statuses
- Draft
- Active
- Stable
- Archived
- Superseded

### Why this matters
This prevents confusion between:
- unfinished work
- current public guidance
- preserved but outdated material

Status should become a visible part of the system over time.

---

## Content Relationships

Content should not live as isolated units.

Every major page should ideally connect to at least one of the following:

- a parent concept
- a related module
- a deeper archive item
- a signal or active note
- a relevant system page

This creates continuity and reduces fragmentation.

---

## Navigation Logic for Content

The content system should support movement between layers.

### Example flow A
`Home -> About -> Manifesto -> Archive`

### Example flow B
`Home -> System -> Project -> Related Archive`

### Example flow C
`Home -> Signals -> Related LAB page -> Archive note`

Pages should help users move deeper without becoming overwhelming.

---

## Publishing Logic

Not every piece of writing should become a public page immediately.

Before publishing content, it should be checked for:

- clarity
- relevance
- structural fit
- alignment with brand core
- whether it belongs to surface, living, or archive layer
- whether it needs metadata or versioning

This reduces premature publishing and system clutter.
